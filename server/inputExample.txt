# Software Requirements Specification (SRS)

## 1. Introduction

### 1.1 Purpose

This document describes the functional and non-functional requirements for **TaskFlow**, a web-based task management application designed for small teams to plan, track, and collaborate on work.

The intended audience includes product owners, developers, testers, and stakeholders.

### 1.2 Scope

TaskFlow allows users to:

* Create and manage projects
* Break projects into tasks
* Assign tasks to team members
* Track task status and deadlines
* Collaborate via comments and notifications

The system will be accessible through modern web browsers.

### 1.3 Definitions, Acronyms, and Abbreviations

* **User**: A registered individual using the system
* **Admin**: A user with elevated permissions
* **Task**: A unit of work within a project
* **Project**: A container for tasks
* **SRS**: Software Requirements Specification

## 2. Overall Description

### 2.1 Product Perspective

TaskFlow is a standalone web application that integrates with third-party authentication providers and email notification services.

### 2.2 User Classes and Characteristics

* **Admin**

  * Manages users and projects
  * Has full system access
* **Team Member**

  * Creates and updates tasks
  * Views assigned projects
* **Viewer**

  * Can view projects and tasks
  * Cannot modify data

### 2.3 Operating Environment

* Web application
* Supported browsers: Chrome, Firefox, Safari, Edge
* Backend hosted on cloud infrastructure

### 2.4 Design and Implementation Constraints

* Must use secure authentication (OAuth 2.0)
* Must comply with basic data protection regulations
* Response time should be under 2 seconds for most operations

---

## 3. Functional Requirements

### 3.1 User Management

**FR-1** The system shall allow users to register using email and password.
**FR-2** The system shall allow users to log in and log out securely.
**FR-3** The system shall allow admins to deactivate user accounts.

---

### 3.2 Project Management

**FR-4** The system shall allow users to create a project.
**FR-5** The system shall allow users to edit project details including name and description.
**FR-6** The system shall allow admins to delete projects.
**FR-7** The system shall allow users to view a list of projects they belong to.

---

### 3.3 Task Management

**FR-8** The system shall allow users to create tasks within a project.
**FR-9** The system shall allow users to assign tasks to one or more team members.
**FR-10** The system shall allow users to update task status (To Do, In Progress, Done).
**FR-11** The system shall allow users to set task deadlines.
**FR-12** The system shall allow users to view tasks assigned to them.

---

### 3.4 Collaboration and Notifications

**FR-13** The system shall allow users to comment on tasks.
**FR-14** The system shall notify users when they are assigned to a task.
**FR-15** The system shall notify users when a task deadline is approaching.

---

## 4. Non-Functional Requirements

### 4.1 Performance

* The system shall support up to 1,000 concurrent users.
* Page load time shall not exceed 2 seconds under normal load.

### 4.2 Security

* User passwords shall be encrypted at rest.
* All communication shall occur over HTTPS.

### 4.3 Usability

* The system shall provide a responsive user interface.
* New users should be able to create a project within 5 minutes of first use.

### 4.4 Reliability

* The system shall have an uptime of 99.5% per month.

---

## 5. Assumptions and Dependencies

* Users have internet access.
* Email notification service availability is required.
* Third-party authentication providers are operational.

---

## 6. Future Enhancements

* Mobile application support
* File attachments for tasks
* Advanced reporting and analytics
